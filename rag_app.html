    Retrieval-Augmented Generation (RAG) System for Academic Research Papers

    Overview

    In the naive RAG, we saw different sections of RAG like indexing, retrieval and generation. However, this is not the end of the world, we can see how these basic module in the naive RAG should be tweaked to cater to a real world application.

    At its core, our system combines dense vector retrieval with generative language modeling. When given a query q and a corpus of documents D, it generates a summary y as follows: y* = arg max P(y | q, R(q, D); θ) (1) Here, R(q, D) represents the set of document chunks retrieved in response to query q, and θ represents the parameters of our generative language model. The system is modular, processing user queries through three distinct pathways: corpus-level summarization, document-specific summarization, and metadata extraction. Each pathway follows a similar basic pipeline (preprocessing, semantic indexing, and response generation) but operates on different data sources and task scopes.     Dataset and Knowledge Base

    To support flexible and accurate query answering, the system combines a large-scale academic corpus with dynamic user inputs. For the background knowledge base, we use the entire arXiv Dataset for Summarization from Hugging Face [13], which contains thousands of arXiv papers paired with their abstracts. This corpus spans a wide range of scientific fields, giving the system a strong foundation of factual, domain-specific knowledge. In addition to this static corpus, users can also upload any number of their own PDF documents, whether they’re unpublished papers, project reports, or drafts. Once uploaded, the system automatically processes these documents and builds a custom vector database on the fly. This makes it possible to perform fast semantic retrieval across both the public corpus and user-specific content, ensuring that responses are grounded in the most relevant and personalized information available. Earlier versions of our system used a smaller prototype knowledge base built from just 10 papers across contrasting domains. This helped us test the system’s ability to handle multi-topic inputs and different writing styles. Now, with both large-scale corpus support and flexible document upload, our system can be scaled for a wide range of use cases.

    Semantic Routing

    A key limitation in our initial system was its tendency to treat all queries the same. Regardless of user intent, the pipeline followed a uniform path, often producing mismatched, incoherent, or irrelevant summaries. To address this, we introduced a semantic routing layer that recognizes the type of question being asked and dynamically routes it to the most appropriate processing module. The current implementation supports three distinct routing paths:
    • Corpus Query Routing: These are broad, open-ended queries (e.g., "What are recent trends in transformer-based vision models?") that require synthesis across multiple papers. For such queries, we apply a multi-document RAG pipeline that retrieves relevant chunks from the entire corpus and generates a role-conditioned summary using a large language model. This route excels at cross-document exploration and thematic synthesis.
    • Metadata Query Routing: These are targeted queries about bibliographic or structural information from specific papers (e.g., "Who are the authors of paper 2?"). A metadata module handles these queries through section-aware PDF parsing and rule-based heuristics to extract fields like title, authors, venue, and publication year. This parsed metadata is stored in a structured index for quick access.
    • Document-Specific Query Routing:strong> When queries reference specific uploaded documents (e.g., "Summarize the results section of paper 3 for a policymaker"), we restrict the retrieval scope to those documents. This approach improves factual alignment and reduces hallucinations by providing the LLM with a focused context window tailored to the user's intent.

    Routing decisions employ a hybrid strategy combining heuristic intent detection with semantic similarity scoring. Given a query embedding zq and a set of prototype intent vectors {zc}c∈C, the selected route c* is defined as: c* = arg max c∈C cos(zq, zc) where C is the set of routing classes and cos(·, ·) denotes cosine similarity. This lightweight formulation enables the system to handle diverse queries while maintaining interpretability and modularity.

    Corpus- and Document-level Modules

    Document Pre-processing and Semantic Chunking

    Using the notations from Equation 1, let D = {d1, d2, ..., dN} be a corpus of N documents or research papers. Each document di is split into sections, which are further divided into paragraphs. Most papers in D contain standard sections like abstract, introduction, methodology, results, and conclusion. These sections vary in their format and content—for example, methods and results typically contain more numerical and mathematical content than the introduction and conclusion. We begin by segmenting each document into text chunks xij: di → {xi1, xi2, ..., xiT} Each xij represents an overlapping chunk of text (usually a paragraph or logical subsection) extracted from document di, designed to capture a self-contained scientific argument, result, or claim. We apply these chunking constraints: • Chunks align with sentence boundaries where possible, using natural language parsing tools • Chunks maintain a 10% overlap as a buffer to ensure coherence and self-containment • The token length of each chunk is limited by a maximum threshold Tmax, where len(xij) ≤ Tmax In our implementation, Tmax is set to 256 tokens.

    Dense Embedding and Retrieval Module

    After segmentation, we convert each chunk x ∈ D into a dense vector using a sentence-level transformer model. We use SciBERT (a BERT-based model trained specifically on scientific text) [14] to capture technical language and domain-specific concepts common in research papers. SciBERT outperforms general-purpose models on our scientific content. We define our embedding function fϕ, where ϕ represents the SciBERT encoder parameters. For any text chunk x and user query q, the vector representations are: zx = fϕ(x), zq = fϕ(q) Since both vectors exist in the same semantic space, we measure their similarity using cosine distance. We retrieve the most relevant content by selecting the top-k chunks based on this similarity: R(q, D) = top−k{x ∈ D : cos(zq, zx)} (2) where cos(a, b) = a · b ∥a∥ ∥b∥ (3) We implement the HNSW (Hierarchical Navigable Small World) indexing strategy [15] from the FAISS (Facebook AI Similarity Search) library [16] to store and search all zx vectors. Unlike our previous flat indexing approach that used brute-force search across the entire embedding space, HNSW creates a multi-layer graph where nodes represent embeddings and edges connect semantically similar points. This structure enables more efficient approximate nearest neighbor (ANN) search, particularly when handling multiple documents. After retrieving the top-k most relevant chunks, we pass them to the generation module, where they combine with the user's query and prompt instructions to produce a role-aware response using a LLM.

    Metadata Module

    Pre-processing

    In addition to answering content-based questions, our system also supports queries that target structured metadata. These include questions like “Who are the authors of this paper?” or “What year was it published?”; queries that don’t require deep semantic reasoning, but instead depend on accurate extraction of key bibliographic fields from the uploaded document. The metadata pipeline begins with document-level pre-processing, where uploaded PDFs are parsed using layout-aware text extraction methods. This stage removes LaTeX artifacts, isolates header and footer regions, and prepares the raw text for field-based extraction.

    Metadata Formatting

    Next, we extract structured fields from each document di to form its metadata record Mi . Each record is defined as: Mi = {titlei , authorsi , venuei , yeari} To populate Mi , we use regular expression-based heuristics along with simple formatting logic. Titles are typically identified by detecting bold, center-aligned text blocks near the top of the document. Authors are parsed using keyword patterns (e.g., affiliations, “and” conjunctions, email addresses), while the venue and year are extracted from footnote regions or known templates. 2.5.3 Metadata Indexing and Query Resolution Once structured, the metadata entries are stored in a key-value format and indexed using FAISS to allow for semantic matching when queries are less exact. Given a metadata query q, such as “Who wrote paper 2?”, the system performs the following steps: • Embed the query using a lightweight sentence encoder: zq = fϕ(q) • Retrieve the top matching metadata record Mi using cosine similarity: M∗ = arg max Mi cos(zq, zMi ) • Extract the relevant field from M∗ , based on a simple query-to-slot mapping function: f(q) ∈ {title, authors, venue, year} Our metadata routing path is designed for speed and accuracy. While we still use a language model to generate the final response in the next step, it’s grounded in structured metadata that’s already been extracted. This helps reduce hallucinations and keeps the model focused on phrasing, not fact-finding, making the system both lightweight and reliable.

    Generative Language Modeling and Role-based Conditioning

    After retrieving relevant context chunks using semantic similarity search, the next stage of our pipeline is language generation; this phase acts on the outputs of all three previous modules (corpus, metadata and user-uploaded pdf). We use a LLM that has been conditioned on both retrieved evidence and user-specific instructions. Using the set of top-k retrieved segments R(q, D) = {x1, x2, . . . , xk} from the previous step, we generate a single, concatenated prompt c, which acts as input to a generative transformer model. We currently use Google’s Gemini 2.0 Flash, because it can support longer contexts, has proven zero-shot summarization capabilities and follows instructions well (which aligns with our goal of role-based conditioning).     User-Provided Papers Mode

    The main goal of this system is to provide accurate answers to various user queries related to academic papers. These queries can range from asking for detailed explanations of concepts in the paper to searching for specific papers based on metadata (like papers published after a certain year). For example, a user might ask:

    • “Can you explain this concept from the paper?”
    • “Which papers published after 2024 discuss X?”

    By handling both content-based and metadata-based queries, the system can meet a wide range of research needs.

    Processing of Contents for the PDF Submitted by the User

    When a user uploads a PDF, the system starts by preprocessing the document. It creates a .json file for each section of the paper. This is important because academic papers can have different structures, and this preprocessing ensures that the system understands each section properly.

    After this, the text is tokenized (broken down into smaller parts) and passed through the embedding model to generate vectorized representations. These embeddings are what allow the system to understand the meaning of different sections of the paper and help it find relevant information when the user asks a question.

    Processing of Metadata for the PDF Submitted by the User

    Along with processing the paper's content, the system also needs to handle the metadata—the information that describes the paper, such as its title, authors, publication date, abstract, and keywords. This metadata allows the system to answer questions like:

    • “Which papers were published after 2024?”
    • “Which papers were presented at a particular conference?”

    The system extracts this metadata by identifying key details such as the file name, title, author(s), publication year, abstract, and keywords.

    However, there is a challenge when dealing with metadata: it is often presented in different formats across various papers. For example, dates might be in a format like yyyymmddhhmmss (e.g., 20170428001908), which needs to be converted into a standard format, like 2017-04-28T00:19:08+00:00. This conversion ensures that the date can be uniformly processed and used for queries.

    Because research papers come from many different sources, there is no fixed format for metadata. This means the system must be flexible enough to handle various ways that metadata might be formatted. For example, conference names, locations, or dates might be listed in different ways, so the system needs special rules to accurately extract and index that information.
